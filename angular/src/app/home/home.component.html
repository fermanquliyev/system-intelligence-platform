About this deployment

SystemIntelligencePlatform — developer reference

Open-source log analysis and incident detection for teams that want full control of their data. Single-tenant, self-hosted stack on PostgreSQL, RabbitMQ, ASP.NET Core, and Angular — no SaaS or tenant model. The repository may still be named system-intelligence-platform; product documentation uses SystemIntelligencePlatform.

Overview

SystemIntelligencePlatform ingests logs, detects anomalies, optionally enriches incidents with an LLM, and pushes live updates to the browser. One deployment per installation — minimal moving parts beyond Postgres, RabbitMQ, and an optional LLM API key.

Features

  • Log ingestion — API key–authenticated HTTP endpoint; events are queued for async processing.
  • Incident detection — Adaptive anomaly detection from recent log patterns.
  • AI-powered analysis — LlmIncidentAiAnalyzer (Google Gemini-compatible API) with automatic fallback to LocalIncidentAiAnalyzer when no API key is set or the call fails.
  • Real-time updates — SignalR broadcasts to authenticated clients.
  • Self-hosted — Typical stack: Postgres, RabbitMQ, API, worker, Angular (e.g. docker-compose up).
  • Advanced intelligence — Copilot chat, full-text log search, timelines, metrics, alert rules, clustering, playbooks, log source config, assignee/merge/PII workflows. See Advanced platform features below.

Advanced platform features

The following capabilities ship with the current codebase. REST shapes follow ABP auto API controllers: base path /api/app plus a kebab-case segment derived from the application service name (e.g. AiCopilotAppService → /api/app/ai-copilot). HTTP verbs and action paths match ABP conventions (e.g. GetAnalysisAsync → GET …/analysis/:incidentId). In examples below, :param denotes a route parameter (replace with a real GUID or id). Most features work offline; outbound LLM calls need AI__ApiKey (or equivalent). After upgrades, run DbMigrator for EF migrations and permission seeds.

Per-feature reference (inputs & outputs)

Each paragraph is one feature. DTO and property names match the C# contracts in SystemIntelligencePlatform.Application.Contracts.

AI Incident Copilot. Service IAiCopilotAppService (/api/app/ai-copilot), permission SystemIntelligencePlatform.Incidents.Copilot. Input → output: GetAnalysisAsync(Guid incidentId) returns CopilotAnalysisDto: RootCauseHypothesis, SuggestedFixSteps, ConfidenceScore (integer 0–100), SuggestedSeverity (IncidentSeverity), optional SeverityJustification, PromptTemplateVersion, FromCache (whether the result was served from cache). PostFollowUpAsync(Guid incidentId, CopilotFollowUpInput input) — input: required Message (string, max 4000 characters); persists conversation turns and returns CopilotMessageDto with IncidentId, Role (User, Assistant, or System), Content, plus creation audit fields. GetConversationAsync(Guid incidentId) returns IReadOnlyList<CopilotMessageDto> for that incident. Orchestration uses versioned prompt templates, cache keys from incident/log context, and IIncidentAiAnalyzer (Google LLM path or local fallback). UI: incident detail Copilot panel; client state via @ngrx/component-store.

Incident assignee. IIncidentAppService.AssignAsync(Guid id, AssignIncidentInput input) (POST /api/app/incident/:id/assign with JSON body), permission SystemIntelligencePlatform.Incidents.Assign. Input: AssignIncidentInput.UserId — nullable Guid (null clears assignee). Output: updated IncidentDto including AssignedUserId and resolved AssigneeUserName when available. Listing endpoints omit incidents that are merged into another (MergedIntoIncidentId not null) so duplicates stay out of the main list.

Incident merge metadata (read model). IncidentDto exposes merge and PII flags for the UI and integrations: MergedIntoIncidentId (nullable) points to the canonical incident when this row is a child; MergedChildIncidentIds lists child ids when this incident is canonical; ContainsPii indicates sensitive content was detected on related data. GetMergedChildrenAsync(Guid canonicalIncidentId) (GET /api/app/incident/merged-children/:canonicalIncidentId) returns IReadOnlyList<IncidentDto> for merged children.

Incident deduplication scan. IIncidentMergeAppService.ScanAndMergeAsync(Guid? applicationId) (/api/app/incident-merge; POST action scan-and-merge with optional applicationId query). Output: int — number of merge operations applied (title/signature similarity heuristics in the application layer). Use after ingest or from an admin workflow; merged incidents are linked in the database and surfaced via IncidentDto and GetMergedChildrenAsync.

PII detection and masking. Type IPiiDetector (implementation PiiDetector, regex-oriented for emails, etc.) flags and masks text at API boundaries. Log search respects LogSearchInput.RevealSensitive: raw text is only returned if the caller is granted SystemIntelligencePlatform.LogEvents.ViewUnmasked; otherwise messages are masked. LogEventSearchItemDto.ContainsPii and IncidentDto.ContainsPii expose the flag to clients. The background worker can set ContainsPii on ingested LogEvent rows and propagate to incidents.

Full-text log search. ILogSearchAppService.SearchAsync(LogSearchInput input) (GET /api/app/log-search/search with query parameters bound to LogSearchInput), permission SystemIntelligencePlatform.LogEvents.Search. Input LogSearchInput extends paging/sorting: Query (optional text), UseFullText (default true — PostgreSQL tsvector / GIN index path), optional ApplicationId, MinLevel (LogLevel), FromUtc, ToUtc, RevealSensitive, plus SkipCount / MaxResultCount / Sorting. Output: PagedResultDto<LogEventSearchItemDto> — each item: Id, ApplicationId, ApplicationName, Level, Message (masked unless allowed), Timestamp, ContainsPii. UI: /log-search.

Saved log queries. Same service, LogEvents.Search where noted. CreateSavedAsync(CreateSavedLogSearchDto input) — input: Name (required, max 128), FilterJson (required, opaque JSON storing filter state, e.g. query + flags). Output: SavedLogSearchDto (Id, Name, FilterJson, audit fields). GetSavedListAsync() → ListResultDto<SavedLogSearchDto>. DeleteSavedAsync(Guid id) → void (deletes only the current user’s saved row; otherwise authorization fails). HTTP: POST /api/app/log-search/saved, GET /api/app/log-search/saved, DELETE /api/app/log-search/saved/:id.

Global incident timeline. IIncidentAppService.GetGlobalTimelineAsync(GetGlobalTimelineInput input), permission SystemIntelligencePlatform.Incidents.Timeline. Input GetGlobalTimelineInput: inherits paging/sorting; filters ApplicationId, Severity, FromUtc, ToUtc, optional TimeScale hint for UI bucketing. Output: PagedResultDto<IncidentTimelineItemDto> — items include Id, Timestamp, Title, Severity, Status, ApplicationName, Kind (e.g. incident). UI: /timeline.

Per-incident root-cause timeline. GetRootCauseTimelineAsync(Guid incidentId) (GET /api/app/incident/:incidentId/root-cause-timeline), Incidents.Timeline. Output: IReadOnlyList<IncidentRootCauseTimelineItemDto> — each item: EventType, TimestampUtc, Title, optional Detail, IsCritical (ordered narrative of logs/events around the incident window). Shown on incident detail as a compact timeline strip.

Metrics ingestion. IMetricAppService.IngestAsync(MetricBatchIngestDto input) (POST /api/app/metric/ingest), permission SystemIntelligencePlatform.Metrics.Ingest. Input: Samples — non-empty list of MetricSampleIngestDto: ApplicationId, Name (metric name, max 128), Timestamp, Value (double), optional TagsJson (max 2000). Output: none (writes MetricSample rows). UI sample button on /metrics posts one synthetic point for testing.

Metric time series query. GetSeriesAsync(GetMetricSeriesInput input) (GET /api/app/metric/series with query-bound properties), permission SystemIntelligencePlatform.Metrics.Default. Input: ApplicationId, Name, FromUtc, ToUtc, MaxPoints (default 500 cap on returned rows). Output: MetricSeriesDto — Name and Points as List<MetricSeriesPointDto> (Timestamp, Value), ascending by time.

Metrics correlation around an incident. GetCorrelationForIncidentAsync(Guid incidentId) (GET /api/app/metric/correlation-for-incident/:incidentId), Metrics.Default. Output: IReadOnlyList<MetricSeriesDto> — for the incident’s application and time window (incident first/last occurrence ± padding), up to several distinct metric Name series with points loaded via the same series query (implementation caps distinct names). UI renders small sparklines per series on /metrics.

Alert rules (CRUD and enablement). IAlertRuleAppService (/api/app/alert-rule), permissions AlertRules.Default (read) and AlertRules.Manage (mutations). GetAsync(Guid id) → AlertRuleDto. GetListAsync(PagedAndSortedResultRequestDto) → PagedResultDto<AlertRuleDto>. CreateAsync(CreateUpdateAlertRuleDto) / UpdateAsync(id, CreateUpdateAlertRuleDto) → AlertRuleDto. CreateUpdateAlertRuleDto: Name (required, max 256), IsEnabled, DefinitionJson (required JSON string), optional SeverityOverride, optional scoped ApplicationId. DeleteAsync(Guid id). SetEnabledAsync(Guid id, bool isEnabled) (PUT …/:id/set-enabled?isEnabled=) → updated AlertRuleDto. AlertRuleDto mirrors persisted fields including audit columns from ABP.

Alert rule evaluation and history. GetHistoryAsync(PagedAndSortedResultRequestDto) → PagedResultDto<AlertHistoryDto> — each row: AlertRuleId, optional ApplicationId, PayloadJson (JSON payload describing the firing), FiredAt, creation audit. EvaluateNowAsync(Guid? applicationId) (POST …/evaluate-now) runs all enabled rules against the last hour of logs per application (single app if applicationId set; otherwise every monitored application), inserts history when thresholds match, returns int fire count. DefinitionJson deserializes to AlertRuleDefinitionModel with optional minErrorsLastHour and/or minWarningsLastHour (JSON property names snake-cased as in the model). UI: /alert-rules.

Adaptive anomaly detection (domain, incident creation path). AnomalyDetectionService.Evaluate(AnomalyMetrics metrics, LogLevel logLevel) returns AnomalyDetectionResult: ShouldTrigger, Reason (None, SpikeDetected, BurstDetected, ImmediateCritical, ZScoreSpike), SuggestedSeverity. Uses 5-minute spike ratio vs hourly baseline, 1-hour burst vs 2× baseline, immediate trigger on Critical log level, new-signature absolute thresholds when baseline is zero, and Z-score on 1-hour volume when baseline > 0 and standard deviation > 0 (evaluated before spike/burst so large statistical deviations can win first). Inputs are aggregate counts in AnomalyMetrics (EventsLast5Min, EventsLast1Hour, EventsLast24Hours, AverageHourlyBaseline, StandardDeviation). The worker uses this when deciding whether to open or update incidents.

Log clustering. ILogClusterAppService (/api/app/log-cluster), LogEvents.Default for list; RunClusteringAsync requires LogEvents.Search. GetListAsync(PagedAndSortedResultRequestDto) → PagedResultDto<LogClusterDto> — ApplicationId, SignatureHash (hash of digit-normalized template), Summary, FirstSeen, LastSeen, EventCount, audit fields. RunClusteringAsync(Guid? applicationId) (POST …/run-clustering) processes a bounded batch of unclustered LogEvent rows, groups by template hash, upserts LogCluster, sets LogEvent.ClusterId, returns int (assignment count). UI: /grouped-logs.

Playbooks. IPlaybookAppService (/api/app/playbook), permissions Playbooks.Default, Playbooks.Manage (create/delete), Playbooks.Run (run and complete steps). GetAsync / GetListAsync → PlaybookDto with Name, Description, TriggerDefinitionJson, IsActive, Steps as List<PlaybookStepDto> (SortOrder, Title, Body). CreateAsync(CreatePlaybookDto) — input: Name, optional Description, TriggerDefinitionJson, Steps as List<CreatePlaybookStepDto> (SortOrder, Title, Body). DeleteAsync(Guid id) removes playbook steps and any runs. RunForIncidentAsync(Guid playbookId, Guid incidentId) (POST …/run-for-incident with query params) → PlaybookRunDto: PlaybookId, IncidentId, Status (InProgress, Completed, Cancelled), StartedAt, optional CompletedAt, RunSteps as List<PlaybookRunStepDto> (StepOrder, Title, IsCompleted, CompletedAt). CompleteRunStepAsync(Guid runId, int stepOrder) marks a step done; when all steps complete, run Status becomes Completed. GetActiveRunAsync(Guid incidentId) (GET …/active-run?incidentId=) → PlaybookRunDto or null. UI: /playbooks.

Log source configuration. ILogSourceConfigurationAppService (/api/app/log-source-configuration), LogSources.Default / LogSources.Manage. DTO LogSourceConfigurationDto: Name, SourceType (File = 0, Http = 1, Syslog = 2), IsEnabled, SettingsJson. CreateAsync(CreateUpdateLogSourceConfigurationDto) / UpdateAsync(id, …) use the same shape for mutable fields. SetEnabledAsync(id, bool isEnabled) (PUT …/:id/set-enabled) toggles without full update. The host registers ILogSourceAdapter implementations (file / HTTP / syslog); shipped stubs can be replaced with real listeners. UI: /log-sources.

Quick start

  1. Clone the repository.
  2. Configure AI (optional). Copy .env.example to .env at the repo root: 
    AI__ApiKey=your-google-ai-studio-key
    Docker Compose reads .env for substitution. If AI__ApiKey is empty, the system still runs using the local analyzer only.
  3. Run the stack 
    docker-compose up --build
  4. Open endpoints (default ports — adjust if yours differ)
    • UI — Angular dev server or container (often http://localhost:4200)
    • API / OpenIddict — host project (often http://localhost:44397)
    • RabbitMQ management — http://localhost:15672 (default guest / guest)
  5. Sign in — Default admin is seeded by ABP; check SystemIntelligencePlatformConsts in the host for development email/password defaults.

Architecture

Angular → API → RabbitMQ (log-ingestion) → Worker → PostgreSQL
                              ↓
                    RabbitMQ (incident-notifications)
                              ↓
                         API → SignalR → Angular
  • API (SystemIntelligencePlatform.HttpApi.Host) — REST, OpenIddict, SignalR hub, optional RabbitMQ consumer for incident notifications.
  • Worker (SystemIntelligencePlatform.BackgroundWorker) — Consumes log messages, persists events, runs detection and AI, publishes incident notifications.
  • Database — EF Core + PostgreSQL; search via DatabaseIncidentSearchService (SQL ILIKE, no Elasticsearch).

Configuration

Override with environment variables (e.g. AI__ApiKey, DataRetention__LogRetentionDays) or a .env file for Docker Compose.

Area Setting Notes
AI AI:Provider, AI:ApiKey, AI:Model Bound to GoogleAiOptions; worker and API use the same shape.
Data retention DataRetention:LogRetentionDays Background job deletes old LogEvent rows (default 90).
Queue RabbitMQ:* Host, port, credentials, virtual host.

Customization

  • Replace the AI provider — Implement IIncidentAiAnalyzer or adapt LlmIncidentAiAnalyzer / ServiceCollectionExtensions.AddLlmIncidentAiAnalyzer for another HTTP API; keep LocalIncidentAiAnalyzer as fallback.
  • Tune detection — AnomalyDetectionService in the domain/application layer defines thresholds and rules.
  • Extend the model — Add entities under Domain, register in SystemIntelligencePlatformDbContext, expose via application services and Angular as usual for ABP.

Local development (without Docker)

  • Restore NuGet packages; set ConnectionStrings:Default and RabbitMQ in appsettings.Development.json.
  • Run the DbMigrator or apply EF migrations to PostgreSQL.
  • Start HttpApi.Host, BackgroundWorker, and ng serve for the Angular app.

Angular UI

Single-instance web UI: dashboard, applications, incidents, search, timelines, metrics, alerting, clustering, playbooks, log sources, and settings. No tenant switcher, tenant headers, or per-tenant routing — one deployment maps to one backend. A route map is listed under Advanced platform features.

Requirements

Node.js 18+, npm (or yarn).

Install

From the angular folder:

npm install

If the solution uses ABP client libraries from the repo root:

abp install-libs

Connect to the API

  • Development: src/environments/environment.ts
  • Production: merged with dynamic-env.json via remoteEnv in environment.prod.ts

Set:

  • apis.default.url — base URL of the host (e.g. http://localhost:44397)
  • oAuthConfig.issuer — usually the same origin as the API
  • oAuthConfig.redirectUri — Angular origin (e.g. http://localhost:4200)
  • oAuthConfig.clientId — must match the OpenIddict client on the server

Optional ABP isolation headers (__tenant, etc.) are stripped by ClearIsolationHeadersInterceptor so stale client state cannot affect requests.

Run & build

Start the host API first, then from angular:

npm start
# or
ng serve
ng build
ng build --configuration production

Regenerate TypeScript API clients after backend changes (from solution root, paths as configured):

abp generate-proxy -t ng

More: ABP Angular UI, Angular CLI.
